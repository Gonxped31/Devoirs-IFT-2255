Cas 3

Cas d'utilisation : Création d'actions
But : Créer une action pour un robot
Préconditions : 1. Avoir un compte Robotix. 2. Avoir les composantes nécessaires pour les actions
Acteurs : Utilisateur (principal), Système (secondaire)
Scénario principal :
1. L'utilisateur se connecte à son compte
2. L'utilisateur choisi l'option de créer les actions à ses robots
3. L'utilisateur choisi le robot qu'il souhaite modifier
4. L'utilisateur choisi le ou les fonctionalités à ajouter à son robot comme par exemple: diffuser des sons, parler, écouter et afficher du texte ou de graphiques.
5. Le système vérifie que l'utilisateur a toutes les composantes nécéssaires. 6. Le système ajoute l'action puis affiche un message de succès Scénarios alternatifs :
5a L'utilisateur n'a pas les composantes nécessaires pour produire les fonctionalités
 5a.1. Le système affiche un message d'erreur
 5a.2. On reprend à l'étape 4
Postconditions : Système enregistre les actions du robot créées par l'utilisateur, l'inventaire de composantes

Cas 4

Cas d'utilisation : Voir les activités maintenues
But : Visualiser les activités qui ont été créées ou qui ont été jointes rejoindre des groupes avec des utilisateurs qui ont des tâches similaires
Préconditions : Avoir un compte Robotix
Acteurs : Utilisateurs (principale), Système (secondaire)
Scénario principal :
1. L'utilisateur se connecte à son compte
2. L'utilisateur choisi l'option de voir les activités
3. Le système affiche les activités maintenues.
4. L'utilisateur sélectionne une activité.
Scénarios alternatifs :
4a. L'utilisateur n'est pas inscrit dans l'activité
 4a.1. Le système ajoute l'utilisateur à l'activité
 4a.2. Le système envoie un message de succès d'ajout
4b. L'utilisateur est inscrit dans l'activité
 4b.1. Le système montre les tâches dans l'activité, ainsi que les robots présent dans l'activité
Postconditions : L'utilisateur sauvegarde les changements, et revient à la page d'acceuil.

Cas 6

Cas d'utilisation : Afficher les métriques de la flotte
But : Visualiser l'état général de la flotte
Préconditions : Avoir un compte utilisateur Robotix
Acteurs : L'utilisateur (principal), Le système(secondaire)
Scénario principal :
1. L'utilisateur se connecte à son compte
2. L'utilisateur sélectionne l'option de voir l'état de sa flotte
3. Le système affiche les métriques dela flotte, par exemple: nombre de robots, robots à "soigner", moyenne de chaque état d'un robot soit: sa vitesse, son niveau de batterie, sa consommation CPU et mémoire.
Scénarios alternatifs :
2a. L'utilisateur n'a pas de robots
Postconditions : L'utilisateur revient à la page d'acceuil

Cas 7

Cas d'utilisation : Gestion de problèmes
But : Gérer des problèmes avec la flotte et/ou un robot
Préconditions : Avoir un compte utilisateur Robotix
Acteurs : Utilisateur (principal), Système (secondaire)
Scénario principal :
1. L'utilisateur se connecte à son compte
2. L'utilisateur choisi l'option de gérer un problème.
3. Le système affiche la liste des problèmes et les propositions d'actions à mener pour les résoudre, et ainsi que ceux qui ont été résolus
4. L'utilisateur confirme les actions à exécuter.
5. Le système résout le problème
Scénarios alternatifs :
4.a Le système affiche les actions à faire pour résoudre le problème
Postconditions : Les modifications apportées par l'utilisateurs sont réenregistré dans la base de données

Tâche 4 : Risques

1. Plusieurs robots, utilisateurs, ou fournisseurs, avec le même nom pourrait créer des erreurs d'affichage
2. Problème d'échelle. Trop d'utilisateurs et de fournisseurs ayant un compte. Gérer les personnels 3. Cross platform 4.

Tâche 5 : Exigences non-fonctionnelles

1. Contraintes de temps. L'application voulue a trop de fonctionalités qui demande un temps de développement qui va au dela de 2 mois.
2. Grand effort de maintenance qui s'étendra fort probablement durant toute la vie du logiciel. Nous n'avons pas la main d'oeuvre à long terme pour maintenir un tel projet.
3. Trop d'utilisateurs en même temps.
