Tâche 3 : Cas d'utilisation

Cas 1

Cas d'utilisation : Inscription d’un client en tant qu' utilisateur.
But : Un client désire créer un compte en tant qu’utilisateur de Robotix.
Préconditions : Posséder une compagnie
Acteurs : Client (principal), Système (secondaire).
Scénario principal
1. Étape 1 : Le client accède à l’interface Robotix.
2. Étape 2 : Le client accède au menu principal.
3. Étape 3 : Le client choisit l’option de création de compte utilisateur.
4. Étape 4 : Le client complète le champ “Profile” en entrant son nom, son prénom, son pseudo, son courriel, son numéro de téléphone et également le nom de sa compagnie.
5. Étape 5 : Le système valide les informations du client et affiche un message de confirmation d’inscription.
6. Étape 6 : Le système affiche le menu d’utilisateur avec les différentes options d’utilisation de robots possibles.
Scénarios alternatifs
5a. Le système rejette certaines informations mal entrées (erreurs de saisie par exemple).
 5a.1. Le logiciel affiche un message d’erreur et demande au client d'entrer les données de nouveau.
 5a.2. Le scénario reprend à l’étape 4.

5b. Le pseudo entré existe déjà.
 5b.1. Le logiciel affiche un message d’erreur et demande au client d'entrer un autre pseudo.
 5b.2. Le scénario reprend à l’étape 4.
Postconditions : Le nouvel utilisateur est enregistré dans la base de données et a maintenant accès aux différentes options d’utilisateur.

Cas 2

Cas d'utilisation : Inscription d’un client en tant que fournisseur.
But : Un client désire créer un compte en tant que fournisseur de Robotix.
Préconditions : Posséder une compagnie.
Acteurs : Client (principal), système (secondaire).
1. Étape 1 : Le client accède à l’interface Robotix.
2. Étape 2 : Le client accède au menu principal.
3. Étape 3 : Le client choisit l’option de création de compte fournisseur.
4. Étape 4 : Le client complète le champ “Profile” en entrant le nom (unique), le prénom, le courriel ainsi que le numéro de téléphone et le nom de sa compagnie.
5. Étape 5 : Le client entre ensuite le type de robot fabriqué, le type de composantes fabriquées et leur capacité de fabrication.
6. Étape 6 : Le système valide les informations du client
 6.1 Le système affiche un message de confirmation d’inscription si les informations entrées sont valides.
7. Étape 7 : Le système affiche le menu fournisseur.
Scénarios alternatifs
6a. Les informations entrées sur le robot et les composantes sont invalides.
 6a.1. Le logiciel informe le client du problème.
 6a.2. Le scénario reprend à l’étape 5.

6b. Le nom du fournisseur existe déjà.
 6b.1. Le logiciel informe le client qu’il est déjà membre.
 6b.2. Le scénario se termine.
Postconditions : Le nouveau fournisseur sera enregistré dans la base de données.
Il sera mis dans la même liste de fournisseurs que les autres fournisseurs ayant les mêmes caractéristiques que lui.
Il sera donc présent dans la liste des fournisseurs affiché lorsque l’utilisateur effectue sa recherche de fournisseur.

Cas 3

Cas d'utilisation : Enregistrement d’un robot.
But : Un utilisateur souhaite enregistrer un nouveau robot.
Préconditions : Posséder un numéro de série (provenant d’un fournisseur) et le type du robot à enregistrer.
Acteurs : Utilisateur (principal), système (secondaire).
Scénario principal
1. Étape 1 : L’utilisateur se connecte à Robotix.
2. Étape 2 : L’utilisateur choisit l’option d’enregistrement de robot.
3. Étape 3 : L’utilisateur entre le numéro de série du robot à enregistrer.
4. Étape 4 : L’utilisateur entre le type du robot à enregistrer.
5. Étape 5 : L’utilisateur entre le nom du robot à enregistrer.
 5.1 Le système vérifie que les informations entrées (type et numéro de série) correspondent bien aux informations fournies par un fournisseur présent dans la base de données des fournisseurs.
6. Étape 6 : Le système affiche un message de confirmation indiquant que le robot a bien été enregistré.
7. Étape 7 : Le système affiche de nouveau le menu utilisateur.
Scénarios alternatifs :
5a. Le numéro de série est invalide (le numéro a été mal saisi).
 5a.1. Le système n’a pas trouvé de robot possédant ce numéro de série.
 5a.2. Le logiciel affiche un message d’erreur et demande au client d’entrer un numéro de nouveau.
 5a.3. Le scénario reprend à l’étape 3.

5b. Le type du robot entré n’est pas valide.
 5b.1. Le système n’a pas trouvé de robot de ce type.
 5b.2. Le logiciel affiche un message d’erreur et demande au client d’entrer un autre type.
 5b.3. Le scénario reprend à l’étape 4.
Postconditions : Le robot est ajouté dans la liste des robots enregistrés pouvant être suivi et pouvant également être assignés à des tâches.

Cas 4

Cas d'utilisation : Créer des tâches
But : Un client désire de créer des tâches qui seront effectués par les robots
Préconditions : Posséder un compte à Robotix
Acteurs : utilisateur (principal), système (secondaire).
Scénario principal :
1. Le client se connecte à Robotix.
2. Le système vérifie le compte du client.
3. Le client se rend au menu principal de l'application.
4. Le client choisit l'option de création de tâche.
5. Le client écrit les détails de la tâche en lui donnant un titre et une description.
6. Le système valide la tâche et affiche un message de confirmation en disant que la tâche a été créée.
Scénarios alternatifs :
2a. Le compte entré n'est pas enregistré sur l'application.
 2a.1. Le système affiche un message d'erreur en disant qu'il n'y a pas de compte enregistré sur l'application.
 2a.2. Le scénario se reprend à l'étape 1.

6a. Le système ne valide pas la tâche.
 6a.1. Le système affiche un message d'erreur et demande au client de récrire à nouveau la tâche en respectant les conditions.
 6a.2. Le scénario reprend à l'étape 5.
Postconditions : Le système a enregistré dans la base de données la tâche, ce qui inclut le titre et la description de cette tâche.

Cas 6

Cas 9

Cas d'utilisation : Afficher l'état des mes robots
But : Permettre à un utilisateur de surveiller l'utilisation de l'énergie et des ressources par un robot.
Préconditions : L'utilisateur doit être connecté au système.
L'utilisateur doit avoir un ou plusieurs robots enregistrés dans le système.
Acteurs : Utilisateur (principal), Système (secondaire).
Scénario principal :
1. L'utilisateur se connecte à son compte sur la plateforme.
2. L'utilisateur sélctionne l'option d'affichage de l'état des robots.
3. L'utilisateur sélectionne le robot dont il souhaite voir l'état.
 3.1 Le sytème met à jour les informations du robot.
4. Le système affiche l'état du robot (position, vitesse, batterie, CPU).
Scénarios alternatifs :
3.1a. Si le système n'a pas de données récentes sur le robot.
 3.1a.1 Le système informe l'utilisateur qu'il n'y a pas de données récentes disponibles.
 3.1a.2 Le scénario se termine.
Postconditions : L'utilisateur a visualisé l'efficacité énergétique et l'utilisation des ressources du robot sélectionné.

Cas 12

Cas d'utilisation : Trouver un fournisseur
But : Un client désire de rechercher un fournisseur dans la liste des fournisseurs enregistrés
Préconditions : Posséder un compte utilisateur Robotix ; Avoir au moins un fournisseur inscrit
Acteurs : utilisateur (principal), système (secondaire)
Scénario principal :
1. Le client se connecte à Robotix.
2. Le système vérifie le compte du client.
3. Le client se rend au menu principal de l'application.
4. Le client choisit l'option de rechercher un fournisseur
5. Le client choisit l'option de filtre facilitant la recherche d'un fournisseur (Ex: par nom, adresse, email, téléphone).
6. Le système valide le choix du fournisseur recherché.
Scénarios alternatifs :
2a. Le compte entré n'est pas enregistré sur l'application.
 2a.1. Le système affiche un message d'erreur en disant qu'il n'y a pas de compte enregistré sur l'application.
 2a.2. Le scénario reprend à l'étape 1.

6a. Le système ne trouve pas le fournisseur recherché par le client.
 6a.1. Le système affiche un message d'erreur et informe au client que sa recherche ne figure pas parmi la liste des fournisseurs inscrits.
 6a.2. Le scénario reprend à l'étape 5.
Postconditions : Le fournisseur est ajouté dans la liste des fournisseurs déjà sélectionnés.

Cas 13

Cas d'utilisation : Mise à jour du catalogue de produits du fournisseur.
But : Permettre aux fournisseurs de mettre à jour leur catalogue de produits.
Préconditions : Le fournisseur est inscrit sur la plateforme, est connecté à son compte.
Acteurs : Fournisseur (principal), Système (secondaire).
Scénario principal :
1. Le fournisseur se connecte à son compte sur la plateforme.
2. Le fournisseur navigue vers la section "Mon catalogue".
3. Le fournisseur sélectionne l'option "Modifier le catalogue".
4. Le fournisseur ajoute un nouveau robot ou une nouvelle composante, ou retire un robot ou une composante qui n'est plus fabriquée.
5. Le fournisseur sélectionne l'option "Enregistrer les modifications".
 5.1 Le système vérifie que l'action faite est possible.
6. Le système valide les modifications et confirme que les modifications ont été enregistrées.
Scénarios alternatifs :
5.1a Le fournisseur tente d'ajouter un produit existant déjà dans le catalogue.
 5.1a.1. Le système affiche un message d'erreur indiquant que le produit est déjà listé et demande au fournisseur de choisir un autre produit.
 5.1a.2. Le scénario reprend à l'étape 4.

5.1b Le fournisseur tente de retirer un produit qui n'est pas dans son catalogue.
 5.1b.1 Le système affiche un message d'erreur indiquant que le produit n'est pas trouvé dans le catalogue et demande de choisir un autre produit.
 5.1b.2 Le scénario reprend à l'étape 4.
Postconditions : Le catalogue de produits du fournisseur est mis à jour avec les nouvelles informations.
Toutes les modifications sont visibles pour les clients et autres utilisateurs de la plateforme.
